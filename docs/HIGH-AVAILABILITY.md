# Vexor — High Availability & Poller Failover

> Operator guide for running Vexor with redundancy: an **active/passive central
> node** and **distributed pollers** for remote-site monitoring.
>
> Read this together with your Vexor disaster-recovery runbook (rebuild
> from zero / backup & restore) and `OPERATIONS.md` (day-to-day).
>
> **Important — what is built-in vs. recommended.** Vexor ships a real
> *distributed poller* feature (documented below from the code). It does **not**
> ship a turnkey clustering product for the central node. The active/passive
> design in section 4 is a **recommended reference architecture** built from
> standard Linux HA building blocks (keepalived, database replication, file
> sync). Sections are labelled **[Built-in]** or **[Recommended practice]** so you
> know which is which.

## 1. Overview

High availability for Vexor covers two **independent** concerns. You can adopt
either one on its own.

* **(a) Central / management node availability.** The management node runs the
  whole stack: the `vexor-api` FastAPI backend, the `vexor-ui` SPA (served by
  nginx), the Naemon monitoring core, MariaDB/MySQL, Keycloak (+ its PostgreSQL),
  InfluxDB for performance data, and Livestatus. If this node dies, the UI/API
  and active monitoring stop. HA here means an **active/passive** pair where a
  standby node can take over a floating address. This is **not** a built-in
  feature — see section 4 for the recommended architecture.

* **(b) Poller / remote-execution availability.** A **poller** is a small agent
  that runs at a remote site and executes checks on behalf of the central node,
  pushing results back. Pollers let you monitor isolated networks (DMZ, branch
  office, customer LAN) without opening inbound connectivity. This **is** a
  built-in feature (`/api/v1/pollers/*`). Pollers have a heartbeat but **no
  automatic check re-assignment** if one dies — see section 5.

```
                        ┌──────────────────────────────┐
        floating VIP ──►│  Central node (active)        │
   (keepalived, §4)     │  nginx · vexor-api · naemon   │
                        │  MariaDB · InfluxDB · Keycloak│
                        └───────────┬──────────────────┘
                replicate / sync    │ (§2, §4)
                        ┌───────────▼──────────────────┐
                        │  Central node (standby)       │
                        └──────────────────────────────┘
                                    ▲
                  POST /me/results  │  GET /me/assigned   (§5, X-Vexor-Poller-Token)
                ┌───────────────────┴───────────────────┐
                │ poller (DMZ)   │ poller (branch site)  │  ← remote agents
                └────────────────┴───────────────────────┘
```

Throughout this document the two central nodes are referred to generically as
`vexor-a` (primary) and `vexor-b` (standby), reachable behind a floating virtual
IP published as e.g. `vexor.example.com`.

## 2. Stateful vs. stateless components

Replicate/share **only** the stateful components. The stateless ones are
reproduced from the RPMs and config and can simply be installed identically on
both nodes.

### Stateful — must be replicated or shared

| Component | Where it lives | Why it is stateful |
|---|---|---|
| **Vexor application DB** (MariaDB/MySQL) | `mariadb.service` data dir (`/var/lib/mysql`) | Hosts, services, host groups, **pollers** (tokens, assignments), users/RBAC, audit log, SLA/BSM config, notification channels, schema. The source of truth the API writes from. |
| **Naemon object config** | `/etc/naemon/vexor/{hosts,services,commands,templates}`, `conf.d/vexor-*`, `_servicedeps.cfg` | Generated by `vexor-api` from the DB; this is what Naemon actually loads. Rebuilt on reload but should be in sync on the standby. |
| **Naemon retention** | `/var/lib/naemon/retention.dat` | Live check state, acknowledgements, downtimes, comments. Lost retention = all states reset to PENDING on failover. |
| **LKG config mirror** | `/var/lib/vexor/naemon-lkg/tree` | Last-known-good config used by the self-rollback guard (see `OPERATIONS.md` §4b). |
| **Performance data** (InfluxDB) | `influxdb.service` data dir | Historical perfdata/graphs. |
| **Keycloak realm + users** | `postgresql.service` (Keycloak's DB) | Identity, SSO/LDAP/MFA config. Only relevant if you use the bundled Keycloak. |
| **`/etc/vexor`** | config dir | `poller_bootstrap.token`, generated `poller.conf`, `logs.env`, branding, welcome-wizard flag, SSH known_hosts for poller deploy. |
| **`/var/lib/vexor`** | state dir | SNMP credentials (`snmp_credentials.json`), LKG tree, misc state. |
| **Custom plugins** | `/opt/vexor/plugins` | Operator-supplied check plugins; also **served to pollers** for sync (`/api/v1/pollers/me/plugins`). |
| **Log data** (optional) | `vexor-victorialogs` data dir | Only if the logs subsystem is in use. |
| **Uploaded files / SSH keys** | `/etc/vexor/ssh/`, registered SSH keys | Keys used by agent/poller deploy. |

### Stateless — install identically on both nodes, no replication

| Component | Unit | Notes |
|---|---|---|
| `vexor-api` (FastAPI) | `vexor-api.service` | Reads DB + Livestatus; holds no durable local state. |
| `vexor-ui` (React SPA) | served by `nginx.service` | Static assets from the RPM. |
| nginx (reverse proxy/TLS) | `nginx.service` | Config from the RPM; terminates TLS for the VIP. |
| `vexor-jobd` (privileged job runner) | `vexor-jobd.service` | Recreates its queue dirs on start. |
| Background evaluators/timers | `vexor-log-alerts-evaluator`, `vexor-perfdata-collector.timer`, `vexor-nightly.timer`, `vexor-db-backup.timer`, etc. | Recompute from the DB; **must only run on the active node** to avoid double work. |

> **Run timers/evaluators on the active node only.** On the standby, leave the
> aux timers and `vexor-api`/`naemon` stopped until failover (keepalived notify
> scripts can start them). Two active Naemon cores writing to the same DB or
> sending duplicate notifications is a misconfiguration.

## 3. Service start/stop order

Use the **real systemd unit names** below. Order matters because `vexor-api`
needs the DB and Livestatus, and Naemon needs its config present.

**Start order (active node coming up):**

```bash
systemctl start mariadb.service        # 1. application DB
systemctl start postgresql.service     # 2. Keycloak's DB (if Keycloak used)
systemctl start keycloak.service       # 3. identity provider (if used)
systemctl start influxdb.service       # 4. perfdata store
systemctl start naemon.service         # 5. monitoring core (creates livestatus.sock)
systemctl start vexor-api.service      # 6. backend (needs DB + livestatus)
systemctl start vexor-jobd.service     # 7. privileged job runner
systemctl start nginx.service          # 8. reverse proxy / TLS front door
# optional log pipeline, if installed:
systemctl start vexor-vector.service vexor-victorialogs.service
```

**Stop order (draining a node) is the reverse:**

```bash
systemctl stop nginx.service
systemctl stop vexor-jobd.service
systemctl stop vexor-api.service
systemctl stop naemon.service
systemctl stop influxdb.service keycloak.service postgresql.service
systemctl stop mariadb.service         # last
```

> If you replicate MariaDB (section 4) the DB stays up on both nodes; in that
> case skip `mariadb.service` in the stop/start cycle and only move the
> stateless + Naemon layer.

## 4. Active/passive central node (recommended practice)

> **[Recommended practice — not a built-in Vexor feature.]** Vexor does not ship
> a clustering daemon for the central node. The following assembles standard,
> well-understood Linux HA components. Test it end-to-end before relying on it.

### 4.1 Building blocks

1. **Floating VIP via keepalived (VRRP).** A virtual IP that always points at
   whichever node is active. Clients, agents and pollers use the VIP hostname
   (`vexor.example.com`), never a node's real address.
2. **Database replication.** Either
   * **MariaDB primary/replica** (asynchronous) — simplest; promote the replica
     on failover; or
   * **Galera cluster** (synchronous, 3 nodes incl. an arbitrator) — no manual
     promotion, but more moving parts. Run **Naemon/vexor-api active on one node
     only** regardless of Galera.
3. **Replicated config & perfdata.** Keep the stateful paths from section 2 in
   sync from active → standby (e.g. a periodic `rsync` or shared/replicated
   storage):
   ```bash
   # Run on the ACTIVE node, e.g. every 1–5 min via systemd timer / cron.
   rsync -aHX --delete /etc/naemon/vexor/        vexor-b:/etc/naemon/vexor/
   rsync -aHX          /var/lib/naemon/retention.dat vexor-b:/var/lib/naemon/retention.dat
   rsync -aHX --delete /etc/vexor/               vexor-b:/etc/vexor/
   rsync -aHX --delete /var/lib/vexor/           vexor-b:/var/lib/vexor/
   rsync -aHX --delete /opt/vexor/plugins/       vexor-b:/opt/vexor/plugins/
   ```
   For InfluxDB perfdata use InfluxDB's own backup/replication, or accept that
   graphs reset to the standby's last sync point. Keycloak's realm rides along
   with PostgreSQL replication.

### 4.2 Keepalived sketch

```conf
# /etc/keepalived/keepalived.conf  (primary = vexor-a)
vrrp_script chk_vexor {
    script  "/usr/bin/curl -fsk https://127.0.0.1/api/healthz -o /dev/null"
    interval 5
    fall     3
    rise     2
}
vrrp_instance VEXOR_VIP {
    state           MASTER          # BACKUP on vexor-b
    interface       eth0
    virtual_router_id 51
    priority        150             # lower (e.g. 100) on vexor-b
    advert_int      1
    virtual_ipaddress { 192.0.2.10/24 }   # the floating VIP (example)
    track_script    { chk_vexor }
    notify_master   "/etc/keepalived/vexor-promote.sh"
    notify_backup   "/etc/keepalived/vexor-demote.sh"
}
```

`vexor-promote.sh` starts the stateless+Naemon layer (section 3) and the aux
timers; `vexor-demote.sh` stops them. This guarantees only the VIP holder runs
an active Naemon core and the background timers.

### 4.3 Failover runbook (planned or unplanned)

> Numbered, runbook-style. Replace `vexor.example.com` / node names with yours.

1. **Confirm the primary is really down** (avoid split-brain). Check from a
   third location:
   ```bash
   curl -fsk https://vexor.example.com/api/healthz   # times out / VIP gone?
   ```
2. **Promote the database** (primary/replica only — skip for Galera):
   ```bash
   # On vexor-b: stop replication and promote the replica to writable primary.
   mysql -e "STOP SLAVE; RESET SLAVE ALL;"
   mysql -e "SET GLOBAL read_only = OFF;"
   ```
3. **Fence the old primary** if it might come back: power it off, or ensure its
   keepalived/nginx/naemon stay stopped, so two nodes never serve the VIP.
4. **Move the VIP.** With keepalived this is automatic once `vexor-a` stops
   advertising; to force it, lower `vexor-a` priority or stop keepalived there.
5. **Start the stack on the standby in order** (section 3). If you use the
   keepalived `notify_master` script this happens automatically:
   ```bash
   systemctl start naemon.service vexor-api.service vexor-jobd.service nginx.service
   ```
6. **Verify** (section 6): health endpoint, Livestatus, UI login, and that
   pollers reconnect (their next `GET /me/assigned` updates `last_seen`).
7. **Re-arm redundancy.** Once the old primary is repaired, re-seed it from the
   new primary and bring it back as the **standby** (do not let it auto-reclaim
   MASTER while stale — use keepalived `nopreempt` or keep its priority lower).

## 5. Distributed pollers & failover (built-in)

> **[Built-in feature]** Implemented in
> `vexor-api/app/routers/pollers_router.py`, with config effects in
> `app/services/naemon_config.py` and housekeeping in `app/services/nightly.py`.
> Managed in the UI under **Settings → Pollers** (`vexor-ui` `pages/settings/Pollers.tsx`).

### 5.1 What a poller is

A poller is a small Python daemon (`vexor-poller.service`) that runs at a remote
site. It **pulls** the list of checks it is assigned, executes the plugins
locally, and **pushes** the results back to the central node, which submits them
to Naemon as **passive** check results. The central node never has to reach into
the remote network.

Master-side endpoints (all under `/api/v1/pollers`):

| Method & path | Auth | Purpose |
|---|---|---|
| `POST /register` | one-time **bootstrap token** | Self-register a poller; returns a permanent token. |
| `GET /` · `POST /` · `PUT /{id}` · `DELETE /{id}` | admin (RBAC) | List / create / update / remove pollers. |
| `POST /{id}/regen-token` | admin | Rotate a poller's token. |
| `POST /{id}/deploy` | admin | Install the agent on a remote host over SSH (uses a distributed SSH key). |
| `GET /{id}/deploy` *(via UI)* / `GET /agent.py`, `/agent.service` | public scripts | Fetch the agent + unit file during bootstrap. |
| `GET /me/assigned` | **poller token** | Poller fetches the host/service checks it must run. |
| `POST /me/results` | poller token | Poller pushes plugin output (rate-limited to 120 req/min/poller). |
| `GET /me/plugins`, `GET /me/plugins/{name}` | poller token | Sync custom plugins from `/opt/vexor/plugins`. |

Pollers authenticate with an `X-Vexor-Poller-Token: <token>` header; only the
SHA-256 of the token is stored on the master.

> **Legacy peer model (also present).** `app/routers/misc_router.py` defines an
> older "Fas 14" notion of a poller as *a separate Vexor instance* with a
> Livestatus port (default **6557**) and a `POST /pollers/{id}/ping` reachability
> probe (a TCP connect that updates `last_seen`/`last_status`). The current,
> actively developed feature is the **agent** model above (it is included last in
> `main.py`, so it owns the `/api/v1/pollers` routes). Treat the Livestatus-peer
> `ping` only as a reachability check, not as a check-distribution mechanism.

### 5.2 How checks are distributed

Assignment is **explicit** or **by host group**:

* **Pinned host** — `hosts.poller_id = <poller id>`. That host (and its services)
  is monitored by that specific poller. In the UI this is the per-host **Poller**
  dropdown (`'' = auto via group`, `0 = master`, `>0 = specific poller`).
* **Host-group filter** — a poller's `hostgroup_filter` is a CSV of host-group
  names. Hosts in those groups **that are not pinned to any poller**
  (`poller_id IS NULL`) are auto-assigned to it.

When a host is pinned to a poller, `naemon_config.py` writes its host/service
objects with:

```
active_checks_enabled    0
passive_checks_enabled   1
check_freshness          1
freshness_threshold      <derived>
```

So the master stops actively checking those objects and instead waits for the
poller to push results. The freshness window is **derived from the check
interval** so a slow check is not flagged stale prematurely: a pinned **host**
uses `600` s, and a pinned **service** uses
`max(600, check_interval_minutes * 120 + 120)` s (roughly two missed cycles plus
slack). Once that window passes with no fresh result, the check is **stale**.

`GET /me/assigned` resolves each assigned host **and service** into a
ready-to-run `command_line`: the service's Naemon command alias (e.g.
`check_rdp`, `check_http!...`, `vexor_exec!...`) is expanded against the command
catalog, `$ARGn$` and host macros (`$HOSTADDRESS$` -> `{address}`) are
substituted, and absolute plugin paths are stripped so the poller can locate the
plugin in its own `plugin_dir`. The agent fills in `{address}`, runs each check
on **its own per-check interval**, and posts results to `POST /me/results`, which
writes `PROCESS_SERVICE_CHECK_RESULT` / `PROCESS_HOST_CHECK_RESULT` lines into the
Naemon external command pipe (opened **non-blocking** so a stalled Naemon reader
can never wedge an API worker).

> **Credentialed checks are skipped on pollers.** Checks that need master-local
> secrets or payloads — agentless WMI (`check_wmi_simple`), agentless SSH
> (`vexor_ssh_check`), or anything referencing `/etc/naemon/keys/`,
> `/etc/naemon/vexor/payloads/` or `/var/lib/vexor/` — cannot run on a remote
> poller. `GET /me/assigned` detects and omits them (reporting a
> `skipped_non_pollable` count), and the host detail page shows an orange
> warning listing the skipped checks. Monitor credential-based checks from the
> master, or move them to an on-host agent (NRPE/NSClient) on the remote site.

### 5.3 Heartbeat & health

Every `GET /me/assigned` and `POST /me/results` call updates the poller's
`last_seen` (and `last_status`). `last_status` is recorded **after** the results
are forwarded and reflects how many were injected, e.g. `ok (12/12 injected)` or
`degraded (0/12; naemon command pipe unavailable (ENXIO))`. The Pollers page surfaces
`last_seen`/`last_status` so you can see at a glance whether a poller is alive.

> There is **no automatic action** taken when `last_seen` goes stale. The
> heartbeat is for **visibility only** — Vexor does not reassign that poller's
> checks to another poller or fall back to active checks on the master.

### 5.4 What happens when a poller goes down

This is the key operational reality:

1. The poller stops calling `POST /me/results`.
2. After the (interval-derived) `freshness_threshold` the pinned host/service
   checks become **stale**. Because active checks are disabled for those objects,
   the master **cannot re-run them itself** — they go to an UNKNOWN/stale state
   and stay there until results resume.
3. `last_seen` on the Pollers page stops advancing.
4. **Nothing is auto-reassigned.** Recovery is operator-driven (below).

So pollers are a **single point of failure for the sites they cover** unless you
build redundancy yourself.

### 5.5 Recommended poller redundancy (N+1, manual/scripted)

> **[Recommended practice]** built on the real reassignment primitives the API
> already exposes. Vexor does not do this automatically.

* **Deploy N+1 pollers per site** so a spare exists, but note that **a host group
  auto-assigns to whichever enabled poller lists that group** — keep `hostgroup_filter`
  values non-overlapping in steady state to avoid duplicate pushes.
* **On poller failure, reassign its hosts** to a healthy poller. Two supported
  primitives:
  * **Delete-with-reassign** (atomic):
    ```bash
    curl -fsk -X DELETE https://vexor.example.com/api/v1/pollers/<dead_id> \
      -H 'Authorization: Bearer <admin-token>' \
      -H 'Content-Type: application/json' \
      -d '{"reassign_to": <healthy_id>}'
    ```
    This repoints every `hosts.poller_id` from the dead poller to the healthy one
    **and re-renders the affected Naemon configs** automatically. Omit
    `reassign_to` (or send `null`) to **unpin to the master** instead, which
    re-enables active checks on the master for those hosts.
  * **Re-point a host group** by editing the healthy poller's `hostgroup_filter`
    (UI or `PUT /api/v1/pollers/{id}`) to include the dead poller's groups.
* **Scripted failover:** poll `GET /api/v1/pollers`, and if a poller's
  `last_seen` is older than your threshold, call the delete-with-reassign (or
  group-repoint) above against a standby poller. This is a thin script you own —
  there is no daemon for it in Vexor.

### 5.6 Housekeeping (built-in)

The nightly job (`nightly.py` → `poller_orphan_cleanup`) resets any
`hosts.poller_id` that points at a poller row which no longer exists (e.g. after
a manual `DELETE` or a race), so hosts don't get stranded on a ghost poller. The
UI flags such hosts with an "orphan poller" badge.

## 6. Health checks

Use these to confirm a node (or the VIP) is healthy after failover:

```bash
# Liveness — unauthenticated, cheap. Always 200 with per-subsystem status.
curl -fsk https://vexor.example.com/api/healthz
# => {"status":"ok","version":"...","checks":{"db":"ok","naemon":"ok"}}

# Simple version/engine probe.
curl -fsk https://vexor.example.com/api/v1/health
# => {"status":"ok","version":"0.1.0","engine":"naemon"}

# Livestatus (local on the node): socket must exist and answer.
test -S /var/lib/naemon/livestatus.sock && \
  printf 'GET status\nColumns: program_start\n\n' | \
  socat - UNIX-CONNECT:/var/lib/naemon/livestatus.sock

# Service layer.
systemctl is-active mariadb naemon vexor-api nginx
```

`GET /api/healthz` reports `db` and `naemon` as `ok`/`degraded` independently, so
a load balancer/keepalived can pull a half-up node out of rotation. Confirm
pollers are healthy on **Settings → Pollers** (or `GET /api/v1/pollers`) — their
`last_seen` should advance within a poll interval after failover.

## 7. Backup & restore

HA is **not** a backup. Replication faithfully copies corruption and bad config
to the standby. Keep real backups and know the restore path:

* **Full rebuild-from-zero, secrets checklist, signing key** →
  your Vexor disaster-recovery runbook.
* The application DB is backed up by `vexor-db-backup.timer`; Keycloak by
  `vexor-keycloak-backup.timer`; config/state self-backup by
  `vexor-selfbackup.timer`. The config auto-rollback (LKG) guard that protects
  Naemon against a bad saved check is described in `OPERATIONS.md` §4b.

## 8. Limitations — what Vexor does NOT do automatically

Set operator expectations correctly:

* **No built-in central-node clustering.** Active/passive (section 4) is a
  recommended assembly of keepalived + DB replication + file sync. Vexor ships no
  cluster manager, no automatic VIP, and no automatic DB promotion.
* **No automatic poller failover.** A poller heartbeat (`last_seen`) is
  visibility only. If a poller dies, its hosts go stale and stay stale until you
  reassign them (section 5.5). There is no auto-reassignment and no automatic
  fallback to active checks on the master.
* **Pinned checks are passive only.** Hosts pinned to a poller have active checks
  disabled on the master (`active_checks_enabled 0`), so the master will not
  silently cover for a dead poller.
* **No cross-node deduplication of notifications/timers.** Two active nodes, or
  the standby's aux timers left running, will double-send. Run the active layer
  on exactly one node (keepalived notify scripts).
* **Retention/perfdata are only as fresh as your last sync.** Async replication
  and periodic `rsync` mean some recent state/graphs can be lost on failover.
  Galera (DB) and InfluxDB native replication reduce this for their data.
* **Split-brain is the operator's responsibility.** Always fence the old primary
  before promoting the standby; never let two nodes hold the VIP or write to two
  diverging databases.
